Operational Plan · Ethereum Archive Node Service

Running an archive node like a service, not a node

An archive node is a read-only RPC service — it holds no keys and signs nothing; signing belongs to the staking side of the stack. Its production concerns are those of any high-value stateful API: availability, the front door, data that only grows, and how fast you can recover it. This plan is organised around those, and marks what this deployment already does against what production adds. Every figure below was measured on the running stack.

What "archive" changes about operations

A full node prunes historical state; an archive node keeps every intermediate state-trie node for every block. Two consequences drive the entire plan.

  • The data only grows, and it is enormous. Mainnet archive is tens of TB. Disk is not a threshold you watch, it is a capacity you continuously plan.
  • You cannot resync it in an incident. Rebuilding a mainnet archive from genesis takes weeks. Recovery must be restore-from-snapshot, never resync — a fact that reshapes backup, HA and upgrades.

Neither is true of a validating full node, which is why an archive service needs its own plan rather than a generic node runbook.

01

Availability & topology

Today the gateway routes to one backend. One archive node down = service down. Every other item here is secondary to fixing that.

              [ ELB / CDN ]        TLS · WAF · global rate-limit · DDoS
                    │
              [ gateway tier ]     method routing · per-key limit · health
              (haproxy, ≥2)
          ┌─────────┼─────────┐
      archive-1 archive-2 archive-3    N replicas, health-checked, drainable
          │         │         │
       [ CL ]    [ CL ]    [ CL ]     each archive EL pairs with its own CL
                    │
           [ snapshot / backup store ]
  • Gap  N archive replicas behind the gateway, so any one can be drained for maintenance or upgrade without downtime. haproxy already supports this — it is server geth2 / geth3 plus health checks — so this is an infrastructure gap, not a config one.
  • Each archive EL pairs with its own CL feeding head updates over the engine API; you do not share one CL across archive ELs.
  • Spread across failure domains — different hosts and AZs, so a zone loss drops one replica, not the service.
  • Done  Backends health-checked with a real eth_blockNumber call, not a TCP ping; an unhealthy backend is removed automatically.

03

Capacity & query management

Archive workloads are bimodal: cheap point reads and expensive replays. A single debug_traceTransaction can pin a core for seconds.

  • Done  The gateway routes debug_/trace_ to a strictly-capped heavy pool (maxconn 2) and point reads to a wider pool (maxconn 64). One expensive query cannot starve the cheap ones — the core resource-isolation problem of archive RPC.
  • Gap  Per-key quotas on heavy methods — trace is a premium tier, not a default.
  • Gap  A caching layer for immutable historical reads — state at a finalised block never changes, so it is perfectly cacheable. Cuts backend load for the common "same historical query, many clients" pattern.
  • Gap  Read-replica scaling — point and heavy pools scale on different replica counts because their cost profiles differ.
04

Data lifecycle — the part unique to archive

The load-bearing decisions of the whole plan live here, because the data is huge and cannot be rebuilt on demand.

Disk growth

Archive disk grows monotonically and fast — plan capacity as a rate, not a level. Done This deployment already alerts predictively: projected time-to-full, not a static "5% left", because the useful warning is "you have N days."

Backup & recovery — the decision everything leans on

  • Gap  Because resync takes weeks, recovery must be restore-from-snapshot: periodic volume snapshots shipped to object storage.
  • Recovery target measured and rehearsed — fresh replica from snapshot to serving in hours, not weeks.
  • Ancient history via era1 files where the client supports it, so it need not ride in every snapshot.
  • Snapshots tested by actually restoring them — an untested backup is a hope, not a plan.

Data integrity

Done The synthetic prober compares the archive node's historical answers against the validating node and pages on divergence. Under a "trust the operator" model this self-check is the only thing between a silently corrupt state DB and a client acting on bad data.

05

Change management

  • Gap  Rolling upgrades: drain → upgrade → re-add → next. The N-replica topology is what makes zero-downtime upgrades possible.
  • Gap  Canary: upgrade one replica, watch its SLIs against the others before rolling the fleet — different versions answering the same queries is also a free correctness check.
  • Gap  Hard-fork readiness: archive nodes must upgrade on schedule or fork onto the wrong chain. Track activation, stage ahead, verify post-fork head agreement.
  • Done  Client diversity: geth for archive, besu for validating — running more than one implementation removes the risk that a single storage bug corrupts the fleet's answers identically.
06

Observability — the strong side of this build

This is where the deployment is ahead of a typical archive service rather than behind.

  • Done  Availability SLO with an error budget (99.9%), measured by a prober through the gateway — user-visible truth, not node self-report — with multi-window burn-rate alerting (fast burn pages, slow burn tickets).
  • Done  Latency SLOs split by cost class — point p95 ≤ 300ms, trace p95 ≤ 10s — because they are different products on one endpoint.
  • Done  Correctness & freshness — divergence pages immediately; tip-lag catches correct answers about stale state.
  • Done  Self-monitoring — the collection pipeline and alert-delivery path are themselves watched; a frozen metric or undelivered alert is caught.
  • Gap  Multi-region probes — measure the network path real users take, not just localhost.

07

Incident response & security

  • Runbooks per alert class, archive-specific ones on top of the general runbook: disk-fill projection breached, tip-lag climbing, correctness divergence, a replica failing health checks.
  • Done  Severity routing already distinguishes page (wake someone) from ticket (working hours), with cause-pages-symptoms-quiet inhibition.
  • Security posture: engine API JWT-authenticated to its CL only; method allowlist at the gateway; TLS everywhere client-facing; network isolation between the serving tier and node internals.
